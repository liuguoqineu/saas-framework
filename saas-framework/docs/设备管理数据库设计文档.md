## 一、设计说明

### 1.1 设计原则

- 所有表采用 InnoDB 引擎，utf8mb4 字符集
- 表名前缀 `device_` 为设备管理模块新表，`biz_repair_` 为扩展现有报修表
- 字段命名采用下划线风格，Java实体类采用驼峰风格（MyBatis Plus 自动映射）
- 所有业务表包含公共字段：`tenant_id`（租户隔离）、`create_time`、`update_time`、`deleted`（逻辑删除）
- 流水表（`device_timeline`、`device_operation_log`）只增不删，不含 `update_time` 和 `deleted`

### 1.2 公共字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT AUTO_INCREMENT | 主键，自增 |
| tenant_id | BIGINT | 租户ID，用于多租户数据隔离 |
| create_time | DATETIME | 创建时间，默认 CURRENT_TIMESTAMP |
| update_time | DATETIME | 更新时间，ON UPDATE CURRENT_TIMESTAMP |
| deleted | INT | 逻辑删除：0-未删除，1-已删除 |

### 1.3 设备与配件追踪机制

| | 设备（item_type=1） | 配件（item_type=2） |
|---|---|---|
| 追踪方式 | 每台唯一追踪，通过 device.id | 按品类数量管理 |
| 采购时 | 创建 device 记录，分配唯一编码 | 不创建 device 记录 |
| 入库时 | 关联 device_id，device 填写 warehouse_name | 只增加库存数量 |
| 在库时 | 每台可查可追踪 | 按品类数量管理 |
| 出库时 | 指定 out_device_id，清空 device.warehouse_name | 只扣减数量 |
| 安装时 | 补充安装位置等信息 | 消耗品，用完即止 |
| 全生命周期 | 一物一码，全程可追溯 | 无需逐一追踪 |

**核心思路：** `device` 表为设备唯一追踪主表（一行=一台设备），`device_inventory` 为品类数量汇总表（一行=一种品类的数量汇总）。两者通过品类信息+warehouse_name 关联。

### 3.6 设备档案主表 `device`

**说明：** 设备终身电子档案，是设备全生命周期管理的核心。设备在采购时即创建档案记录并分配唯一编码，入库时填写 warehouse_name，出库时清空 warehouse_name，安装时补充安装信息。device 表为设备唯一追踪主表（一行=一台设备），device_inventory 为品类数量汇总表。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| id | BIGINT AUTO | PK | - | 主键 |
| device_code | VARCHAR(30) | Y | - | 设备唯一档案编码，唯一 |
| device_name | VARCHAR(100) | Y | - | 设备名称 |
| brand | VARCHAR(50) | N | NULL | 品牌 |
| model | VARCHAR(100) | N | NULL | 型号 |
| spec | VARCHAR(100) | N | NULL | 规格 |
| category | VARCHAR(50) | N | NULL | 设备分类 |
| purchase_order_id | BIGINT | N | NULL | 关联采购单ID（采购时填写） |
| purchase_item_id | BIGINT | N | NULL | 关联采购明细ID（采购时填写，追溯采购源头） |
| stock_in_order_id | BIGINT | N | NULL | 关联入库单ID（入库时填写） |
| stock_out_order_id | BIGINT | N | NULL | 关联出库单ID（出库时填写） |
| warehouse_name | VARCHAR(100) | N | NULL | 当前所在仓库名称（入库时填写，出库/安装后清空） |
| install_location | VARCHAR(200) | N | NULL | 安装位置（安装时填写） |

| install_file | VARCHAR(500) | N | NULL | 安装调试资料附件URL |
| accept_photo | VARCHAR(500) | N | NULL | 验收照片URL |
| status | TINYINT | Y | 0 | 状态：0-待入库，1-待安装，2-在用，3-维修中，4-停用，5-报废 |
| remark | VARCHAR(500) | N | NULL | 备注 |
| tenant_id | BIGINT | Y | - | 租户ID |
| create_time | DATETIME | Y | CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | Y | CURRENT_TIMESTAMP | 更新时间 |
| deleted | INT | Y | 0 | 逻辑删除 |

**索引：**
- `uk_device_code` (device_code) — 唯一索引
- `idx_device_name` (device_name)
- `idx_status` (status)
- `idx_warehouse_name` (warehouse_name)
- `idx_install_location` (install_location)
- `idx_purchase_order_id` (purchase_order_id)
- `idx_purchase_item_id` (purchase_item_id)
- `idx_tenant_id` (tenant_id)

**设备状态流转：**
```
采购创建 → 待入库(0) → 入库 → 待安装(1) → 出库/安装 → 在用(2)
                                                          ↓
                                                     故障报修 → 维修中(3)
                                                          ↓
                                                     维修完成 → 在用(2)
                                                          ↓
                                                     整机更换/停用 → 停用(4)
                                                          ↓
                                                     报废 → 报废(5)
```

## 五、数据规则

### 5.1 出入库不可删除

- 所有出入库记录采用逻辑删除（deleted=1），数据库层面禁止物理删除
- 权限控制：删除操作需高级权限

### 5.2 一物一码全程溯源

- 每台设备在采购时即创建 device 记录并分配唯一编码 `device_code`
- 通过 `device_timeline` 记录完整时间线，实现全程可追溯
- 设备追踪链路：采购创建 → 入库（填warehouse_name）→ 出库（清warehouse_name）→ 安装 → 在用 → 维修 → 报废

### 5.3 出库自动扣减库存

- 出库操作通过数据库事务保证原子性
- 出库前校验库存是否充足，不足则拦截
- 事务内同步更新 `device_inventory.total_qty` 和 `stocked_out_qty`
- 设备出库时同时清空 `device.warehouse_name`

### 5.4 维修更换关联出库耗材

- 维修工单通过 `has_replacement` 标识是否更换配件
- 更换配件时通过 `replacement_id` 关联更换记录主表
- 更换明细子表通过 `stock_out_order_id` 关联出库单，实现耗材管控

### 5.5 操作日志留痕

- 所有增删改操作通过 AOP 切面自动记录到 `device_operation_log`
- 日志记录操作人、操作时间、操作内容、IP等信息
- 支持按模块、操作人、时间范围查询

### 5.6 并发控制

- 出库扣减库存：使用数据库行锁 + 原子更新SQL（`WHERE total_qty >= #{quantity}`）
- 入库增加库存：使用数据库行锁 + 原子更新SQL
- 设备状态变更：使用数据库行锁保证一致性
