## 二、功能测试点及修复指引

### 1. 登录与 Token 获取
**测试步骤**  
- 正确登录 `admin/123456`，应返回 `token` 和 `userInfo`（包含 `permissions` 数组）。  
- 错误密码应返回 401，错误信息 `用户名或密码错误`。  
- 使用错误 token 请求任何需认证接口，应返回 401。

**常见问题**  
- 密码加密未用 BCrypt，或数据库中密码明文导致匹配失败。  
- JWT 秘钥配置错误或过期时间未设置。  
- 返回的 `userInfo` 缺少 `permissions` 字段。

**修复方案**  
- 检查 `UserServiceImpl` 登录逻辑：使用 `BCryptPasswordEncoder.matches`。  
- 检查 `JwtUtil`：签名秘钥一致（`application.yml` 中配置）。  
- 登录成功后从数据库查询用户权限列表（角色关联的权限 code），组装到返回体。

---

### 2. 租户隔离（核心）
**测试步骤**  
- 使用租户A的管理员登录，新增一条学生记录（`name=张三`）。  
- 直接查询数据库 `biz_student`，确认该记录的 `tenant_id` 等于租户A的ID。  
- 使用租户B的管理员登录，查询学生列表，应看不到租户A的“张三”。  
- 使用超级账户 `admin` 登录，查询学生列表，应能看到所有租户的学生。  
- 使用租户A的普通员工（`emp_a1`）登录，查询学生列表，也只能看到租户A的数据。

**常见问题**  
- MyBatis-Plus 多租户插件未配置或配置错误，导致查询不带 `tenant_id` 条件。  
- 插入学生时未自动填充 `tenant_id`（`MetaObjectHandler` 未生效或未从 `TenantContext` 获取）。  
- 超级账户也被强制加上了 `tenant_id = 0` 条件，导致查不到任何业务数据。

**修复方案**  
- 检查 `MybatisPlusConfig`：  
  - `TenantLineInnerInterceptor` 中 `TenantLineHandler` 的 `getTenantId()` 返回当前用户的 `tenantId`，若为 `0` 则返回 `null` 表示不添加条件。  
  - `ignoreTable` 方法让 `sys_tenant` 等不需要租户隔离的表跳过。  
- 检查 `TenantContext` 是否在每个请求开始时从 Token 中解析并设置（可使用拦截器 `HandlerInterceptor`）。  
- 插入时在实体类的 `@TableField(fill = FieldFill.INSERT)` 方法中自动赋值 `tenantId`。

---

### 3. 权限继承与限制（租户管理员给员工分配角色）
**测试步骤**  
- 超级账户登录，创建一个角色 `租户主管`，赋予权限 `student:list, student:add, student:edit`。  
- 将 `租户主管` 角色分配给租户A的管理员（`test_aadmin`）。  
- 租户A管理员登录，尝试创建一个新角色 `学生专员`，只勾选 `student:list` 和 `student:add`（均在自身权限内），应成功。  
- 租户A管理员尝试创建角色 `全能专员`，勾选 `student:delete`（但自己并没有 `student:delete` 权限），后端应返回 403 或 400，提示“权限超出范围”。  
- 租户A管理员给普通员工 `emp_a1` 分配角色，角色列表不应出现 `全能专员`（若之前未创建成功则无所谓），只应出现权限不超出自身权限的角色。  
- 租户A管理员试图将 `student:delete` 权限分配给已有角色（如修改角色），应同样被拦截。

**常见问题**  
- 后端没有校验租户管理员分配/创建角色时的权限范围。  
- 角色列表查询时没有过滤：租户管理员看到的角色包含了自己无权操作的角色。  
- 前端未根据后端返回的可选角色列表做限制（但后端必须兜底）。

**修复方案**  
- 在 `RoleService` 的 `saveOrUpdate` 方法中，获取当前登录用户的所有权限 code 集合，若当前用户不是超级账户，则检查待保存角色的所有权限 code 是否全部属于该集合，否则抛出 `BusinessException(403, "权限超出范围")`。  
- 在 `UserService` 给用户分配角色时，同样校验新角色的权限是否在当前用户的权限范围内。  
- 查询角色列表时，租户管理员应只看到自己创建的角色（`tenant_id=当前租户ID`）以及公共角色（如果需要），但不应该看到其他租户角色；并且不能看到权限超过自己的角色（可以后置过滤，但推荐在后端直接过滤：只返回那些权限集合是自己的权限集合子集的角色）。

---

### 4. 学生管理 CRUD 与权限按钮
**测试步骤**  
- 使用无 `student:add` 权限的账户（如 `emp_a1` 只配了 `student:list`），登录前端，学生管理页面不应显示“新增”按钮（`v-permission` 指令作用）。  
- 直接调用 `POST /api/student` 接口（用 Postman 携带该用户的 token），应返回 403。  
- 使用有 `student:add` 权限的账户（租户管理员），能正常新增，且数据库记录 `tenant_id` 正确。  
- 修改、删除类似测试。  
- 分页查询接口：带 `name` 和 `studentNo` 模糊搜索，应正确过滤。

**常见问题**  
- 后端权限注解 `@RequirePermission` 未生效（AOP 切面未配置）。  
- 前端 `v-permission` 指令未注册或未根据 `store` 中的权限列表判断。  
- 搜索参数后端接收不到（参数名不一致，如 `studentNo` vs `student_no`）。  
- 逻辑删除未配置，删除操作直接物理删除了数据。

**修复方案**  
- 实现 `PermissionAspect`，用 `@Around` 从当前用户信息中获取权限列表，对比方法上的 `@RequirePermission` 值。  
- 前端定义指令 `permission.js`，在 `main.js` 中注册，指令逻辑为 `hasPermission` 函数检查 `store.state.user.permissions` 是否包含所需权限。  
- Controller 中分页参数使用 `@RequestParam(required = false) String name` 等，与前端传参名一致。  
- 实体类 `Student` 上配置 `@TableLogic` 逻辑删除字段。

### 5. 租户管理（仅超级账户）
**测试步骤**  
- 超级账户登录，访问租户管理页面。  
- 创建新租户，填写名称、代码（如 `test_c`）。系统应自动生成管理员 `test_cadmin`，随机密码，并在创建成功后弹窗显示用户名和明文密码。  
- 数据库中 `sys_tenant` 表新增一条记录，`sys_user` 表新增租户管理员，其 `tenant_id` 为新租户 ID，`role_id` 为某个预设的“租户管理员角色”（此角色应拥有租户管理员应有的全部权限，由超级账户事先配置）。  
- 禁用租户：修改状态为禁用后，该租户下所有员工应无法登录（登录时检查 `user.status` 或 `tenant.status`）。

**常见问题**  
- 创建租户时未自动创建租户管理员，或管理员密码明文存储未加密。  
- 租户管理员角色权限不正确（比如缺少管理员工、创建角色的权限）。  
- 禁用租户后，该租户下的用户仍能登录（未做租户状态检查）。

**修复方案**  
- `TenantService.createTenant` 中需要：  
  - 生成 `tenantId`，保存 `sys_tenant`。  
  - 创建角色“租户管理员”（如果不存在），分配合理权限（如 `tenant:*`, `user:manage`, `role:manage` 等）。  
  - 创建用户，设置 `tenant_id` 为新的租户ID，`role_id` 为租户管理员角色，密码 BCrypt 加密。  
- 登录时，除了校验用户名密码，还要检查 `user.status` 以及关联的 `tenant.status`，若租户被禁用则返回“账户已被禁用”。

---

### 6. 员工管理（租户管理员）
**测试步骤**  
- 租户A管理员登录，创建员工，填写用户名、姓名、角色（只能选择权限不超出管理员权限的角色）。  
- 创建成功后，数据库中 `sys_user` 记录的 `tenant_id` 自动为租户A的ID。  
- 租户A管理员修改员工角色，同样不能分配权限超出的角色。  
- 租户A管理员重置员工密码为 `123456`，员工可用该密码登录。  
- 租户A管理员删除员工，应逻辑删除（`deleted=1`）。  
- 租户B管理员登录，看不到租户A的员工列表。

**常见问题**  
- 创建员工时未自动填充 `tenant_id`。  
- 角色下拉列表未过滤，显示了其他租户的角色或权限超出的角色。  
- 重置密码时未重新 BCrypt 加密。

**修复方案**  
- 在 `UserService.addEmployee` 中，通过 `TenantContext.getCurrentTenantId()` 设置 `user.setTenantId(...)`。  
- 查询角色列表时，条件为 `tenant_id=当前租户ID`，并且若当前用户不是超级账户，还需进一步过滤：只返回那些权限集合是当前用户权限集合子集的角色（否则租户管理员可能把自己无权使用的角色分配给员工）。  
- 重置密码：`user.setPassword(bCryptPasswordEncoder.encode("123456"))`。

---

### 7. 角色管理界面与权限树
**测试步骤**  
- 超级账户登录，进入角色管理。  
  - 新增角色，选择权限树部分权限，保存后再次编辑，应回显已选权限。  
  - 删除角色时，若该角色已被用户绑定，应提示“角色已使用，无法删除”或级联清空用户的角色。  
- 租户A管理员登录，进入角色管理，只能看到自己创建的角色（`tenant_id = 租户A ID`），不能看到其他租户的角色。  
  - 创建角色时，权限树中只应显示当前用户拥有的权限（前端可禁用无权节点，后端必须校验）。  
  - 租户A管理员无法修改或删除租户B的角色。

**常见问题**  
- 权限树接口返回了所有权限，前端未做限制，导致租户管理员可以为角色分配自己无权使用的权限（后端会拦截，但体验不好）。  
- 角色列表未按租户隔离，租户管理员看到了超级账户的角色或其他租户角色。  
- 删除角色时未检查是否被用户引用。

**修复方案**  
- 后端 `GET /api/permission/tree` 应接收当前用户，若用户不是超级账户，则只返回其权限范围内的权限树（递归过滤）。  
- `GET /api/role/page` 根据当前用户 `tenant_id` 返回对应租户的角色（超级账户返回所有租户角色 + `tenant_id=0` 的平台角色）。  
- 删除角色前，检查 `sys_user` 表中是否有 `role_id` 等于该角色，若有则抛出异常。

---

### 8. 逻辑错误与边界情况
- **租户管理员不能给自己降权**：假设租户管理员 A 只有 `student:list` 权限，他无法创建或分配任何包含其他权限的角色，这符合预期。但初始创建租户时，管理员应被授予一个完整权限的角色（超级账户提前定义好）。  
- **超级账户创建租户时，允许指定管理员角色**：可选，若不指定则使用默认租户管理员角色。  
- **多租户插件与逻辑删除插件的执行顺序**：确保逻辑删除条件也正确带上租户ID，避免跨租户误删。  
- **事务管理**：创建租户（插入租户表、用户表）需加 `@Transactional`。

## 四、修复任务优先级

1. **P0（阻断性）**：登录功能、租户隔离、权限校验（否则框架无法演示）。  
2. **P1（核心业务）**：学生管理完整 CRUD、员工管理、租户管理。  
3. **P2（体验完善）**：前端按钮权限、角色管理树形展示、密码重置、租户禁用逻辑。  
4. **P3（文档与测试）**：补充单元测试、完善 `TUTORIAL.md`。

## 五、如何快速修复（给 AI 的再指令）

你可以选择将某一类问题单独提取，重新交给 AI，例如：

> “请修复当前项目中的租户隔离问题：MyBatis-Plus 多租户插件未生效，导致跨租户数据泄露。要求：  
> - 配置 `TenantLineInnerInterceptor`，从 `TenantContext` 获取当前租户 ID，若为 0 则不追加条件。  
> - 在拦截器 `TenantInterceptor` 中从 Token 解析用户并设置 `TenantContext`。  
> - 修改所有插入操作，自动填充 `tenant_id`。  
> 请给出修改后的文件代码。”

按模块逐个修复效率更高。
